# 项目内存与学习循环

> 🌐 **Languages:** [English](../../en/explanation/project-memory-and-learning-loop.md) · [Русский](../../ru/explanation/project-memory-and-learning-loop.md) · **简体中文**

Veles 的决定性特性在于它能按项目 **记忆** 并 **学习**。本页解释这种内存是什么，以及学习循环如何让它保持有用。

## 内存是一份结构化产物

项目内存存放于 `<project>/.veles/` —— `memory.db`（SQLite，事实来源）外加一棵人类可读的 `.veles/memory/` 树（渲染后的洞见视图、会话摘要、提案、系统操作日志）。它 **独立于你的内容**，在任何布局（wiki、notes 或 bare）下都以相同方式工作。它不是聊天记录的倾倒，而是一组结构化的层：

- **会话日志** —— 每次对话，每轮一行，并建立全文索引。
- **规则** —— 智能体应当遵循的简短祈使句（`format`、`do`、`don't`、`preference`），注入到稳定的系统提示中。
- **洞见** —— 从会话中提炼出的经验教训。SQL 行是权威的（召回、老化与去重都基于它运行）；同时会向 `.veles/memory/insights/` 渲染一份 markdown 视图，供人类阅读和导出。
- **项目树映射** —— 一份经过缓存、带语义标签的文件映射，让智能体读取 3–5 个相关文件，而不是整棵树。
- **技能与工具注册表** —— 带遥测数据（使用/成功/错误计数），供排序和去重使用。

参阅 [项目布局](../reference/project-layout.md#project-memory-velesmemorydb) 中的表清单。

## 召回：精简的上下文，按需拉取

`AGENTS.md` 刻意保持精简。当你提问时，Veles 只拉取相关的内容：匹配的历史轮次（全文 + 可选的向量重排序）、适用的规则和洞见，以及项目树映射评分最高的文件。这让每次模型调用聚焦且廉价，而不是把一切都倾倒进去。

## 学习循环

经验通过三种机制变为持久知识：

### 洞见 —— 捕获教训
一次运行之后，提取器会寻找值得记住的东西：明确的"记住 X" / "永远不要 Y"反馈，以及工具错误→恢复的模式（一次失败之后跟着一次修复）。它将这些提炼为洞见和规则，从而避免重复同样的错误。

### 策展器 —— 整合会话
策展器将较旧的会话提炼为持久内存：始终生成 SQL 洞见和规则；此外，当项目的布局启用了 wiki 引擎时，还会生成一个 `wiki/sessions/` 页面。它在空闲/轮后定时器上运行，或通过 `veles curate` 按需运行。

### 做梦 —— 后台维护
`veles dream`（以及守护进程在空闲时）会提取洞见、对技能和洞见去重、建议晋升，并（在 wiki 布局下）对 wiki 进行 lint —— 在不阻塞你的前提下让内存保持新鲜。加上 `--include-consolidation` 可进行更深入的 LLM 处理。

## 上下文压缩

长对话通过一个滑动窗口压缩器保持在模型上下文上限之内：当内存中的历史越过某个 token 阈值时，中间部分会被（由一个廉价的路由模型）总结，并替换为指向保存在 `.veles/memory/sessions/` 中摘要的指针。完整历史始终保留在 `memory.db` 中 —— 只有内存中的窗口被压缩，因此在磁盘上是无损的。

## 为什么这很重要

由于内存是结构化的，且循环持续运行，一个 Veles 项目 **用得越多就越有用** —— 它会学习你的约定、避免重复错误，并将答案根植于它实际见过的内容之上。
