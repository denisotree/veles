# Обзор архитектуры

> 🌐 **Языки:** [English](../../en/explanation/architecture.md) · **Русский**

Эта страница объясняет, что Veles *представляет собой* и как его части сочетаются
друг с другом, чтобы остальная документация обрела смысл. За авторитетным
продуктовым видением обращайтесь к `VISION.md` в корне репозитория.

## Замысел проектирования

Veles намеренно **минималистичен и чисто декомпозирован** — модули с единственной
ответственностью, без god-файлов. Он работает по принципу **local-first**: вы
запускаете его на каталоге своей машины, и там же он хранит собственную
структурированную память.

## Пять опор (ядро)

Всё, что есть в ядре, служит одной из пяти задач:

1. **Проектная память** — структурированный артефакт (отдельный от вашего
   контента), хранящий лог сессий, выученные правила/инсайты, карту файлов проекта
   и реестры навыков/инструментов с телеметрией. См. [проектную память и цикл обучения](project-memory-and-learning-loop.md).
2. **Цикл обучения** — куратор, экстрактор инсайтов и «сны» (dreaming), которые
   поддерживают память в актуальном состоянии и превращают опыт в переиспользуемые правила.
3. **Многоагентная оркестрация** — менеджер, который декомпозирует задачу и порождает
   специализированных воркеров. См. [многоагентную оркестрацию](multi-agent-orchestration.md).
4. **Протокол провайдеров** — единый интерфейс поверх множества LLM-бэкендов
   (облако, локальные, делегирование CLI). См. [providers](../reference/providers.md).
5. **Минимальный набор инструментов и навыков** — небольшой стартовый набор, который
   **накапливается** по мере того, как Veles пишет собственные инструменты и
   формализует повторяющиеся процессы в навыки. См.
   [навыки и инструменты](skills-and-tools.md).

## Всё остальное — опциональный модуль

Шлюзы/каналы, демон, планировщик, TUI, vision/STT — все они **подключаемы** и
загружаются только при использовании. Veles стартует с минимумом и расширяется по
требованию, так что простой `veles run` остаётся простым.

## Как протекает ход (turn)

```
your prompt
   │
   ▼
context: AGENTS.md (small) + on-demand recall from project memory
   │
   ▼
agent loop  ──►  provider (routed per task)  ──►  tool calls
   │                                               │
   │            (trust ladder gates sensitive tools)
   ▼
response  ──►  saved to memory  ──►  learning triggers (insights, curator)
```

Контекстный файл (`AGENTS.md`) намеренно держится небольшим; вспомогательные знания
(страницы wiki, карта файлов проекта, релевантные прошлые ходы) подтягиваются **по
требованию**, а не вываливаются заранее.

## Где хранится состояние

- `<project>/.veles/` — память этого проекта, конфигурация, локальные навыки/инструменты.
- `~/.veles/` — пользовательская глобальная конфигурация, межпроектные навыки/инструменты, кэши, доверие.
- `<project>/AGENTS.md`, `wiki/`, `sources/` — ваш контент (раскладка LLM-Wiki).

См. [структуру проекта](../reference/project-layout.md).

## Мультипроект в одном цикле

Один цикл агента обслуживает множество проектов. Каждый проект получает свой каталог
с собственным контекстом и памятью; `AGENTS.md` симлинкуется на
`CLAUDE.md`/`GEMINI.md`, чтобы внешний CLI, запущенный там, видел тот же контекст.
См. [несколько проектов](../how-to/multi-project-and-subprojects.md).

## Поверхности

- **CLI** (`veles run`, `veles add`, …) — одноразовое и скриптовое использование.
- **TUI** (`veles tui`) — интерактивный REPL с [режимами выполнения](modes.md).
- **Демон + каналы** — безголовое API, Telegram, задачи по расписанию.

Все три приводят в действие один и тот же базовый цикл агента.
