# Как подключать внешние MCP-серверы

> 🌐 **Языки:** [English](../../en/how-to/external-mcp-servers.md) · [简体中文](../../zh-CN/how-to/external-mcp-servers.md) · [繁體中文](../../zh-TW/how-to/external-mcp-servers.md) · [日本語](../../ja/how-to/external-mcp-servers.md) · [한국어](../../ko/how-to/external-mcp-servers.md) · [Español](../../es/how-to/external-mcp-servers.md) · [Français](../../fr/how-to/external-mcp-servers.md) · [Italiano](../../it/how-to/external-mcp-servers.md) · [Português (BR)](../../pt-BR/how-to/external-mcp-servers.md) · [Português (PT)](../../pt-PT/how-to/external-mcp-servers.md) · **Русский** · [العربية](../../ar/how-to/external-mcp-servers.md) · [हिन्दी](../../hi/how-to/external-mcp-servers.md) · [বাংলা](../../bn/how-to/external-mcp-servers.md) · [Tiếng Việt](../../vi/how-to/external-mcp-servers.md)

Veles — это [MCP](https://modelcontextprotocol.io/)-**клиент**: он может
подключаться к внешним MCP-серверам и предоставлять их инструменты агенту так, как
будто они встроенные (GitHub, документация библиотек, веб-поиск, ваши собственные
сервисы, …).

## Настройка сервера

Добавьте блок `[mcp.servers.<name>]` в `<project>/.veles/config.toml` (или в
пользовательский глобальный `~/.veles/config.toml`). Имя `<name>` должно
соответствовать `[A-Za-z0-9][A-Za-z0-9_-]{0,31}` — оно становится частью имени
каждого инструмента. Поддерживаются три транспорта: `stdio` (по умолчанию),
`http`, `sse`.

| Ключ | Транспорт | По умолч. | Назначение |
|---|---|---|---|
| `transport` | — | `"stdio"` | `stdio` \| `http` \| `sse` |
| `command` | stdio (обязат.) | — | исполняемый файл для запуска — **только программа, без аргументов** |
| `args` | stdio | `[]` | список аргументов, по одному токену на элемент |
| `env` | stdio | `{}` | доп. окружение для подпроцесса (поверх унаследованного) |
| `url` | http/sse (обязат.) | — | эндпойнт сервера |
| `timeout_s` | — | `120` | бюджет на один вызов инструмента |
| `connect_timeout_s` | — | `30` | бюджет на первичное подключение |
| `enabled` | — | `true` | `false` — оставить запись, но не подключаться |

Строковые значения в `command`, `args`, `env` и `url` подставляют `${VAR}` из
окружения (несуществующая переменная превращается в пустую строку с предупреждением)
— не храните секреты в самом файле.

> **`command` vs `args`.** Veles запускает программу напрямую (без shell), поэтому
> исполняемый файл и его аргументы — **раздельные** поля. Пишите
> `command = "npx"`, `args = ["-y", "pkg"]` — **а не** `command = "npx -y pkg"`.

### stdio (локальный подпроцесс)

```toml
[mcp.servers.github]
transport = "stdio"
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
env = { GITHUB_TOKEN = "${GITHUB_TOKEN}" }
```

Сервер, который вы запускаете сами, настраивается так же — укажите `command`/`args`:

```toml
[mcp.servers.mytools]
transport = "stdio"
command = "python"
args = ["-m", "my_mcp_server"]
```

### Сервер, которому нужен API-ключ (context7)

[Context7](https://context7.com) отдаёт актуальную документацию библиотек.
Передавайте ключ аргументом, чтобы `${VAR}` держал его вне файла:

```toml
[mcp.servers.context7]
transport = "stdio"
command = "npx"
args = ["-y", "@upstash/context7-mcp", "--api-key", "${CONTEXT7_API_KEY}"]
```

```bash
export CONTEXT7_API_KEY=...   # затем запускайте veles
```

### http / sse (удалённый)

```toml
[mcp.servers.search]
transport = "http"            # streamable HTTP; для SSE-эндпойнта используйте "sse"
url = "https://mcp.example.com/mcp"
```

> **Пока без кастомных заголовков.** Транспорты `http`/`sse` отправляют только
> `url` — Veles не может прикрепить заголовок `Authorization`. Для удалённого
> сервера, которому нужен ключ, предпочитайте его `stdio`-вариант (например, через
> `npx`) с ключом в `args`/`env`, либо эндпойнт, принимающий ключ в URL.

## Скрытие отдельных инструментов

Задайте `[mcp] disabled_tools` — таблицу, сопоставляющую каждому серверу список имён
инструментов, которые нужно пропустить:

```toml
[mcp]
disabled_tools = { github = ["delete_repository"], search = ["raw_query"] }
```

## Просмотр и проверка

```bash
veles mcp list              # все настроенные серверы: транспорт, статус, число инструментов
veles mcp test github       # подключиться к одному серверу и вывести его инструменты
```

`veles mcp list` всегда завершается с кодом 0 — это инспектор, а не health-gate.
`veles mcp test` возвращает 1 при неудачном подключении и 2 при неизвестном имени
сервера.

## Как появляются инструменты

После настройки серверы монтируются **автоматически** при следующем `veles run` /
запуске TUI / демона — отдельного флага «включить MCP» нет, переключатель — это сам
факт наличия конфига. Каждый инструмент попадает в обычный реестр как
`mcp_<server>_<tool>` и вызывается агентом как любой встроенный. Их схемы
санитизируются (ограничения имени/длины, удаление управляющих символов), чтобы
недоверенный сервер не мог внедриться в промпт. Подсказки инструментов отображаются
на лестницу доверия: деструктивные инструменты всегда требуют подтверждения,
инструменты только для чтения запускаются без запроса, остальные проходят обычный
поток [доверия](security-and-permissions.md) — выдайте постоянное разрешение через
`veles trust set`, если не хотите, чтобы спрашивали каждый раз.

## Обработка сбоев

Сервер, к которому не удалось подключиться — отсутствует `command`, неверный `url`
или любая невалидная запись — логируется как предупреждение и пропускается. Он
никогда не блокирует запуск или агента. Перезапустите `veles mcp list`, чтобы
увидеть статус и ошибку.
