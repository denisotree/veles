# VISION: Veles

Veles — LLM-агентская система для knowledge work с компаундирующей памятью, иерархической мультиагентной декомпозицией, прозрачной автономией и открытыми интерфейсами. Работает поверх любого проекта, накапливает знания и инструменты со временем, остаётся под полным контролем пользователя. Минималистичное, чисто декомпозированное ядро с модульной архитектурой.

Миссия: дать одному человеку или маленькой группе агента, который растёт вместе с их проектами и со временем требует всё меньше сопровождения, не теряя предсказуемости и приватности.

---

## 1. Аудитория и сценарии

Veles ориентирован на knowledge workers, не только на разработчиков. Целевая аудитория — опытные пользователи терминала, которые ведут несколько долгоживущих проектов: индивидуальная разработка, аналитика, исследования, подготовка документации, обработка данных. Команды — позже, как естественное расширение через shared exports.

Veles одинаково работает в двух режимах:
- поверх существующего проекта — изучает структуру, строит проектную память и (опционально) wiki, помогает развивать;
- с нуля — создаёт проект по запросу, ведёт через дизайн, реализацию, верификацию и итерации.

Scope не ограничен SDLC: любой проект, который можно описать как "идея → артефакт", подходит. Артефакты — код, документы, аналитические отчёты, датасеты, презентации, дизайны.

## 2. Differentiators

Veles стоит на четырёх опорах одновременно. Ни одна по отдельности не уникальна; их сочетание — да.

1. **Hierarchical multi-agent orchestration без "telephone game".** Главный агент-менеджер декомпозирует задачу, спавнит специализированных worker-агентов (explorer, writer, advisor, custom roles), сам не пишет финальный ответ — синтезирует через отдельного writer'а. Промежуточные результаты воркеров доходят до синтезатора в исходном виде, а не как пересказы через цепочку. Конкуренты либо линейны, либо теряют точность на каждой передаче контекста. См. §5.3.

2. **Компаундирующая память и петля самообучения.** Veles ведёт собственную проектную память — структурированный артефакт, отдельный от пользовательских файлов: лог общения, правила работы из обратной связи, карта структуры проекта, реестры доступных tools и skills. Память persistent, переживает сессии, постоянно дополняется. Curator консолидирует завершённые сессии, lint ищет противоречия и устаревшее, telemetry влияет на ранжирование. Конкуренты не сохраняют такого уровня контекста между сессиями. См. §5.1.

3. **Tools и skills как accumulating capability.** Veles сам пишет себе tools (чистые python-скрипты с best practices, наследуемые) и формализует повторяющиеся процессы как skills. Реестры на двух уровнях — project-level и user-level (`~/.veles`), с явным промоушеном между ними. Импорт внешних tool — через системный skill `tool_installer`. Skills могут наследоваться и ссылаются на tools с конкретными аргументами. Никто из существующих агентов не предлагает такой модели роста инструментария. См. §5.4, §5.5.

4. **Backend-агностичность и ансамбли моделей.** Любой LLM-провайдер: openrouter, anthropic, openai, gemini напрямую через API + claude/gemini CLI через подписку (delegation в JSON-stream режиме). В рамках одного провайдера — ансамбли: разные модели на разные типы задач по критериям, advisor-tier1 валидирует ключевые решения base-моделей. См. §5.7.

Модульность ядра и open interfaces — пятая опора (§4 и §6). LLM Wiki в стиле Karpathy — дефолтный layout пользовательского контента, не core-принцип (§5.2).

Между проектами знания по умолчанию **не** смешиваются. Шаринг — только через явный экспорт/импорт.

## 3. North Star и метрики ценности

Главный сигнал ценности через год использования — **автономия и self-improvement**. Veles сам пишет себе скиллы и инструменты, сам консолидирует память, сам предлагает декомпозицию задач и проектов, сам выбирает модель под задачу. Пользователь со временем вмешивается реже в routine и больше в стратегию.

Поддерживающие метрики:
- глубина проектного контекста: "быстрее спросить Veles про свой проект, чем лезть в код";
- экономия токенов и времени за счёт ансамблей, cli-делегации и кэша структуры проекта;
- рост собственного инструментария: число agent-generated tools и skills, доля переиспользований;
- переносимость: проекты-шаблоны можно публиковать и переиспользовать.

## 4. Архитектурные принципы

- **Минимальное ядро.** Цикл агента-менеджера, оркестратор саб-агентов, проектная память, learning loop, провайдер-протокол, базовый набор tools и skills. Всё, без чего Veles не запускается.
- **Всё остальное — модули.** Каналы коммуникации, gateway-интеграции, scheduler, observability, расширенные memory-провайдеры, обработка PDF/изображений, web-fetch, специализированные skills/tools — отдельные пакеты с манифестом.
- **Память отделена от пользовательских файлов.** Veles ведёт собственную проектную память в `<cwd>/.veles/` (SQLite + структурированные файлы): лог, инсайты, карта структуры, реестры. Пользовательский контент проекта (wiki, sources, код, документы) живёт в корне проекта по выбранному layout. Память работает независимо от выбранного layout.
- **Песочница CWD + user-global ресурсы.** Для агента существует только текущий проект (CWD с подпроектами) и user-global часть `~/.veles` (skills, tools, config, реестры). Другие проекты пользователя — невидимы. Симлинки наружу не разрешены.
- **Изоляция проектов.** Все project-specific артефакты (память, telemetry, skills, tools, schema, контент) живут в самом проекте. `~/.veles/projects/` хранит только реестр (slug → path, last-activity), не контент.
- **Writable-зоны LLM декларативны.** Конкретные writable-зоны зависят от выбранного пользователем layout и декларируются в `AGENTS.md`. В дефолтном LLM Wiki layout LLM пишет только в `wiki/` и `<cwd>/.veles/`; `sources/` read-only.
- **Local-first.** Всё хранилище локально (SQLite, файлы). Cloud-зависимости — opt-in через отдельные модули с подтверждением включения.
- **Open interfaces.** Провайдер-протокол, hook-points модулей, формат скиллов и tools, агент-протокол — публичные, документированные, версионированные.

## 5. Ядро возможностей

### 5.1 Проектная память Veles

У Veles есть собственная проектная память — структурированный артефакт, отдельный от пользовательских файлов. Её цель: чтобы агент не переоткрывал на каждом запросе одно и то же и постоянно обучался на обратной связи.

В памяти живут:
- **Лог общения** — все turn'ы агента и пользователя, persistent в SQLite + FTS5. По нему агент отвечает на простые вопросы "что я говорил/решал раньше" без спавна воркеров.
- **Правила и инсайты** — как лучше отвечать на типичные запросы пользователя, как форматировать вывод, что делать и чего не делать. Накапливаются из явной обратной связи ("больше так не делай", "вот так — отлично") и автоматического insight extraction по итогам сессий.
- **Карта структуры проекта** — кэш файловой иерархии и смысловых блоков. Когда пользователь задаёт вопрос — агент сразу читает релевантные файлы, а не сканирует весь проект на каждом запросе. Экономия контекста и времени.
- **Реестры tools и skills** — что доступно (user-level из `~/.veles`, project-level из `<cwd>/.veles`, builtin), описания, аргументы, телеметрия использования (use_count, success_rate, last_used).
- **Реестр runtime-сессий** — daemon- и TUI-сессии проекта (см. §7.3): их настройки, статус, история запусков и soft-delete-флаг. История всех сессий, когда-либо запускавшихся для проекта (включая удалённые), — вход для curator/dreaming активных daemon-сессий.

Память находится в `<cwd>/.veles/` — SQLite-таблицы плюс структурированные markdown-файлы для журналов и инсайтов. Curator работает непрерывно: компактирует завершённые сессии, ищет противоречия, дубликаты, stale-claims, orphan-инсайты. Telemetry skills/tools влияет на ранжирование при дублях и на кандидатов на архивацию.

Память — core. Она работает независимо от того, как пользователь решил организовать содержательную часть проекта (см. §5.2).

### 5.2 LLM Wiki как default layout

Дефолтный шаблон организации **пользовательского контента** проекта — LLM Wiki в стиле Karpathy:
- Три слоя: `sources/` (raw, read-only для агента), `wiki/` (LLM-generated, writable), schema (`AGENTS.md`).
- Три операции: `ingest`, `query`, `lint`.
- Index в `INDEX.md`, append-only journal в `LOG.md`.

Это **опция**, не требование. Пользователь может описать кастомный layout в `AGENTS.md` (свои директории, свои правила записи, своя структура) — проектная память Veles из §5.1 работает поверх любого выбранного layout. При `veles init` дефолтно создаётся LLM Wiki; пользователь может выбрать другой layout или построить свой.

Операции `ingest`/`query`/`lint` **не входят в core CLI** и не зашиты в TUI. Они реализованы как **skills из layout-pack `llm-wiki`**, который подключается автоматически при выборе wiki-layout. Для кастомного layout пользователь определяет свои операции собственными skills (например, `obsidian-import` для Mind Palace или `record-day` для journaling-layout) — core не делает предположений о том, как именно хранится контент.

Wiki-страницы — markdown в файловой системе. Insight extraction (см. §5.1) может писать в wiki как отдельные страницы с cross-references, если выбран wiki-layout; для кастомных layout — пишет в раздел, который пользователь декларировал writable.

### 5.3 Hierarchical multi-agent orchestration

Любая задача пользователя решается через иерархическую декомпозицию. Общий паттерн:

1. **Менеджер понимает и декомпозирует.** Главный агент-менеджер читает запрос, определяет — это простой вопрос по уже накопленному в проектной памяти контексту (отвечает сам без спавна) или сложная задача. Для сложной — решает, сколько worker-агентов нужно и с какими skills. Даже задача "найди в моих файлах X и напиши ответ" спавнит как минимум одного explorer и одного writer — менеджер сам не пишет финальный ответ.

2. **Интерактивное уточнение при развилках.** Если задача недостаточно понятна или есть значимые развилки в принятии решений, менеджер формулирует вопросы с вариантами ответов. Опрос интерактивный: вопросы задаются по одному, переключение вариантов клавишами вверх/вниз, печать ответа только если пользователь выбрал последний пункт (свободный ввод). Этот же UX-паттерн используется во всех точках clarification (см. §7).

3. **План с checkbox'ами.** Менеджер пишет план: входные данные, список саб-агентов, их skills, ожидаемые результаты, зависимости. Агенты могут запускаться последовательно (когда новый агент ждёт результат предыдущего) или параллельно — например, один explorer изучает локальные файлы, второй ищет в интернете через web-fetch, третий через MCP arxiv.org. План видим пользователю как checkbox-список.

4. **Менеджер только координирует.** Менеджер не пишет финальный ответ сам — для синтеза (статьи, кода, ответа пользователю, нового tool) создаётся отдельный writer-агент с подготовленными входными данными. Advisor-агент может быть отдельным пунктом плана (валидация ключевых решений) или вызываться одним из исполнителей в checkpoint'е по своему усмотрению.

5. **Мини-отчёт перед завершением.** До закрытия задачи менеджер пишет короткий отчёт: что было сделано, почему именно так, с какими сложностями столкнулись. Отчёт уходит в проектную память как инсайт для будущих похожих задач.

Принципиальное правило: **никакого "telephone game"**. Промежуточные артефакты воркеров попадают к синтезатору в исходном виде, а не как пересказы через менеджера. Реализовано как чисто декомпозированный слой в core Veles.

### 5.4 Tools — accumulating capability

Tool — это исполняемая единица: python-скрипт, shell-команда, MCP-сервер, внешний бинарь.

**Три origin'а**:
- **Builtin** — поставляются с core Veles (CRUD файлов в CWD, shell, базовый поиск). Минимум для запуска.
- **Agent-generated** — для повторяющейся задачи Veles может написать python-скрипт и сохранить в `<cwd>/.veles/tools/`. Скрипт пишется с best practices (clean, типизированный, документированный) и достаточно универсальный, чтобы для похожей будущей задачи можно было унаследоваться от него или вызвать с другими аргументами. Информация о скрипте — имя, назначение, аргументы, где использовался — записывается в реестр tools в проектной памяти.
- **Manual install** — пользователь указывает директорию или executable и говорит "это новый tool". Вызывается системный skill `tool_installer` — он читает код и документацию, копирует артефакты в `<cwd>/.veles/tools/`, обновляет реестр, опционально пишет wrapper-страницу с описанием использования.

**Два уровня хранения**:
- **Project-level** — `<cwd>/.veles/tools/`. Видны только в этом проекте. Дефолт при создании.
- **User-level** — `~/.veles/tools/`. Доступны во всех проектах пользователя.

**Промоушен**: если tool, написанный для одного проекта, нужен в другом, пользователь (или агент по запросу) переносит код в `~/.veles/tools/`. Обновляются реестры на обоих уровнях. В дальнейшем код используется и дорабатывается на пользовательском уровне.

**Наследование**: новые скрипты могут унаследоваться от существующих (импорт, классовое наследование, композиция) — для группы похожих задач не пишется новый tool с нуля. Это и есть смысл best-practices при изначальном написании: tool сразу проектируется как переиспользуемая база.

### 5.5 Skills — формализованные процессы

Skill — это описание процесса: последовательности шагов, инструментов, проверок для типового рабочего сценария. Skills оформлены как структурированные документы (`SKILL.md` + frontmatter с телеметрией) и индексируются в реестре skills проектной памяти.

- **Создаются автоматически.** В процессе работы Veles распознаёт повторяющиеся процессы и предлагает их формализовать как skill (с одобрением пользователя). Пользователь может создавать skills и вручную.
- **Реестры на двух уровнях**: project-level (`<cwd>/.veles/skills/`) и user-level (`~/.veles/skills/`). Промоушен — как у tools.
- **Наследование.** Может быть базовый skill (общий процесс) и дочерний (ссылается на весь базовый или его части, добавляет специфику). Это позволяет переиспользовать workflow без копирования.
- **Skills ссылаются на tools.** В описании skill указывается какие tools с какими аргументами должны быть вызваны на каждом шаге — это формализует связь "процесс → инструмент".

При коллизии имени побеждает project-level (проектный override общего).

### 5.6 Двумерная декомпозиция знаний

- **Горизонтальная (между проектами)**: пользователь может вести несколько проектов параллельно, переключаясь между ними. `~/.veles/projects/` ведёт лёгкий реестр (slug, путь, last-activity). Внутри одной запущенной сессии агент видит только активный проект — данные других проектов недоступны. Шаринг знаний между проектами — только через явный экспорт/импорт (см. §9).
- **Вертикальная (внутри проекта)**: проект может содержать иерархию подпроектов в собственной файловой структуре (`<cwd>/<subproject>/`). Корневой проект ведёт index подпроектов; каждый подпроект имеет собственный контекст и подгружается по релевантности запроса. Эта иерархия доступна агенту в рамках одной сессии.
- Декомпозицию инициирует агент: при росте контекста или памяти видит смысловые кластеры и предлагает выделить подпроект. Пользователь одобряет.

### 5.7 LLM backend

- **Провайдеры**: openrouter, openai, anthropic, gemini (прямые API), claude-cli и gemini-cli (через подписку, JSON-stream). Открытый адаптер-протокол для добавления новых.
- **Ансамбли**: внутри провайдера — несколько моделей на разные типы задач. Конфигурация — typed config (TOML) глобально + natural-language override в AGENTS.md проекта (override побеждает при конфликте).
- **Advisor pattern**: tier1-модель валидирует ключевые решения base-моделей (план, архитектура, финальный ответ). Advisor вызывается как отдельный саб-агент в hierarchical orchestration (см. §5.3) — отдельным пунктом плана или внутри исполнителя в checkpoint'е, не оборачивает каждый turn.

### 5.8 Мультимодальность

- **Ядро**: текст и структурированные данные (markdown, JSON, CSV, TSV).
- **Модули**: PDF, изображения (OCR или vision-модель), аудио, прочие бинарные форматы. Модуль либо поднимает локальную утилиту, либо использует мультимодальный LLM. Установка — с подтверждением.

## 6. Модульная экосистема

- **Репозиторий**: курируемый official-реестр (`github.com/<veles>/modules`, `github.com/<veles>/skills`) с review от maintainer'ов. Любой git URL тоже допустим, но при установке Veles явно показывает warning "unreviewed source".
- **Манифест**: каждый модуль — директория с `module.toml` (имя, версия, entrypoint, требуемые хуки, минимальная версия Veles).
- **Hook-points**: pre/post-turn, pre/post-tool_call, on_session_start/end, on_subagent_spawn, schedule_tick. Pre-tool-call поддерживает veto.
- **Установка**: `veles add module|skill <git|path>`. По умолчанию требует подтверждения. После установки — в trust-режиме на уровне per-операция (см. §8).
- **Trusted и autopilot**: явные команды `veles trust ...` и `veles autopilot --until <ts>` ослабляют подтверждения для конкретного источника или временного окна.

## 7. Поверхности взаимодействия

Развиваются последовательно:

1. **CLI ad-hoc** (MVP): `veles run "..."`, `veles project ...`, `veles skill|module|tool add|remove`, `veles curate` (работа над проектной памятью §5.1, не над layout). Single-shot и REPL. Layout-specific операции — например, `ingest`/`query`/`lint` для LLM Wiki — **приходят как skills** из соответствующего layout-pack и вызываются через `veles run <skill>` или напрямую агентом, а не как core CLI verbs.
2. **TUI** (next): rich-интерфейс в терминале с историей сессий, панелями проектной памяти (лог, правила, инсайты, реестры tools/skills), управлением jobs. Layout-specific панели (обзор wiki при LLM Wiki layout, или произвольная панель из кастомного layout-pack) **подключаются опционально** через skills, не зашиты в core TUI. Основной интерфейс на этапе после MVP.

   **Требования к UX-унификации TUI** (зафиксированы для рефакторинга):
   - **Единый popup-компонент.** Все модальные блоки (pickers, confirmations, trust ladder, approval, выбор варианта при clarification) рендерятся через один контейнер с консистентным layout: заголовок → тело → список опций. Текущее сосуществование PickerScreen и ComposerPrompt с разной логикой сводится к одному компоненту.
   - **Поле ввода = универсальный input-модуль.** Composer в обычном режиме принимает текст с историей и slash-completion; когда модель задаёт вопрос (clarification из §5.3, trust ladder, approval), **тот же компонент** переключается в interactive question mode: список вариантов с навигацией клавишами вверх/вниз, выбор Enter'ом, последний пункт — свободный ввод (печать активируется только при выборе). Это соответствует универсальному паттерну опроса из §7.
   - **Native выделение, копирование, вставка.** Текст из chat log можно выделить мышью и скопировать через OS clipboard без переключения режимов; вставка из буфера (текст и изображения) — стандартными хоткеями. Никаких отдельных mode-toggle для select.
   - **Команды-инспекторы как dedicated slash-команды.** Подсчёт токенов и текущий объём контекста доступны не только в StatusBar chip, но и через `/tokens` (in/out, per-session, per-turn) и `/context` (текущий размер, лимит модели, состав по типам сообщений). Команда `/status` даёт сводку: model, mode, session, провайдер, autopilot, jobs.
   - **Reasoning виден в инспекторе.** `ctrl+i` раскрывает инспектор, который стримит reasoning агента (Anthropic thinking blocks, OpenAI/OpenRouter reasoning) как typed-события, а не только индикатор «thinking…». На провайдерах без reasoning поведение graceful — остаётся «thinking…».
   - **Ошибки и таймауты — first-class.** Все exceptions, таймауты провайдера и отменённые turn'ы пишутся в (а) persistent error-log проектной памяти и (б) инспектор — не только в chat log, который скроллится. Каждый turn **кооперативно отменяем**: прерывание (Ctrl+C) останавливает провайдерский вызов, процесс выходит чисто, без зависаний.
   - **Интерактивная сессия — runtime проекта.** `veles` без аргументов — это runtime-сессия проекта (наравне с daemon-сессиями), регистрируется в проектной DB и видна в едином session-manager (см. §7.3).

3. **Daemon + внешние каналы** (важный next-step после MVP): Veles как фоновый процесс в системе с открытым API. Поверх daemon — модульные каналы: Telegram-бот, Slack, web UI, generic HTTP/WS. Любой фронт интегрируется через тот же API.

   **Daemon-сессии: несколько на проект.** Daemon привязан к каталогу/проекту. В одном проекте может быть запущено несколько daemon-сессий с разными настройками (в том числе разные model/provider). Каждая daemon-сессия — именованная сущность с собственным lifecycle: **create / start / stop / restart / delete**. Декларативные настройки сессии (model, provider, host, port, mode) живут в `config.toml` (`[daemon.<name>]`) и **перечитываются при restart**; runtime-состояние, история запусков, логи и soft-delete-флаг — в проектной DB. Delete помечает сессию `deleted` (скрыта в TUI), но запись и история сохраняются и используются для dreaming активных daemon-сессий. Интерактивная TUI-сессия (§7.2) — это runtime-сессия того же реестра с `kind = tui`; daemon-сессии — `kind = daemon`. Единая иерархия: **runtime-сессия ⊃ conversation-контексты (по каналу/потоку) ⊃ turns**.

   **Daemon как шина данных.** К одной daemon-сессии подключаются несколько каналов. Daemon выступает шиной: у каждого канала собственный независимый conversation-контекст, но все каналы обращаются к одной daemon-сессии с её фиксированными настройками (model/provider). Канал добавляется к **конкретной запущенной daemon-сессии** через wizard (универсальный паттерн опроса §7); есть команды управления каналами (list/add/remove). Конфигурация каналов декларативна в `config.toml` и поддерживает несколько типов (telegram, generic; slack и далее — модулями).

   **Channels-parity с TUI** (требование ко всем каналам, включая Telegram):
   - Любой канал реализует **все slash-команды** из TUI через native интерфейс канала (для Telegram это `setMyCommands` + handler'ы, для Slack — slash-commands API, для web — UI buttons). Live typing — единственное исключение, оно остаётся TUI-only.
   - **Inline keyboards / interactive buttons** для всех сценариев, где TUI показывает popup: clarification от менеджера, trust ladder, approval, выбор варианта в опросе. В каждом канале это конкретный native механизм (Telegram inline keyboards, Slack block actions и т. д.).
   - **Управление через канал**: переключение модели, mode, провайдера, toggle autopilot — доступно через slash-команды и/или кнопки, не требует возврата в TUI или CLI.
   - **Команды-инспекторы**: `/tokens`, `/context`, `/status` работают одинаково во всех каналах. Каждый канал отдаёт сводку в своём формате.
   - **Режимы агента через slash-команды**: `/goal <task>` (long-running цикл) и `/dream` (consolidation) — те же, что в TUI. Доступны во всех каналах.
   - **Multimodal input**: voice → STT, photo → vision (через соответствующие модули из §5.8). Канал, который физически поддерживает медиа, обязан их принимать; неподдерживаемые типы возвращают понятную ошибку, не silently drop.

**Универсальный паттерн интерактивного опроса.** Везде, где Veles нуждается в clarification от пользователя — менеджер при декомпозиции (§5.3), trust ladder при чувствительной операции (§8), выбор layout при `init`, разрешение конфликта при миграции памяти — используется один и тот же UX: вопросы задаются **по одному**, варианты переключаются клавишами **вверх/вниз**, последний пункт — **свободный ввод** (печать активируется только при его выборе). Это снимает когнитивную нагрузку и делает поведение Veles предсказуемым во всех каналах: в TUI это ListView внутри унифицированного popup'а, в Telegram — inline keyboard на сообщении, в web — buttons-row, но семантика одна.

**First-run experience**: минимальный wizard при первом запуске — выбор языка интерфейса, LLM-провайдера и ключа, имя первого проекта, выбор layout (LLM Wiki по умолчанию). Дальше — REPL с подсказками. Без длинных туров; концепты раскрываются по мере использования.

## 8. Безопасность и автономия

- **Песочница**: только CWD активного проекта (с подпроектами) и user-global часть `~/.veles` (skills, tools, config, реестры). Данные других проектов из реестра не доступны без явного переключения. Всё остальное вне поля видимости агента.
- **Trust ladder per-операция**: при каждом запросе на чувствительную операцию агент показывает 4 опции — "разрешить разово / всегда в этом проекте / всегда везде / отказать". Дальнейшие подтверждения зависят от выбора. UX опроса — как в §7.
- **Autopilot**: `veles autopilot --until <ts>` — временный всеобщий байпас подтверждений до даты. Все автономные действия пишутся в журнал проектной памяти для последующего ревью.
- **Always-confirm операции**, даже в trust/autopilot:
  - отправка данных во внешние сервисы и API (кроме выбранного LLM);
  - удаление файлов и любые деструктивные операции;
  - установка новых модулей, скиллов и tools из сети;
  - подключение нового канала к daemon-сессии (выставляет проект наружу через внешний сервис) и выдача его кредов;
  - изменения вне границ активного проекта (другие проекты, `~/.veles`, OS).
- Часть разрешений можно зафиксировать как дефолтные на уровне проекта.
- Настройки daemon-сессии (model/provider) фиксируются при её запуске из `config.toml` и неизменны в течение жизни сессии (см. §7.3); смена — через restart с обновлённым конфигом.

## 9. Переносимость знаний

Память по умолчанию изолирована per-project. Шаринг — только через явный экспорт.

Два режима:
- **Full export** — bit-for-bit бандл проекта (всё содержимое `<cwd>/.veles/` — проектная память, реестры, telemetry, project-level skills/tools — плюс пользовательский контент проекта). Назначение: бэкап и миграция между своими машинами.
- **Sanitized template** — schema + wiki (или любой контент в writable-зонах) + project-level skills + tools, без логов общения, sessions, sources и PII; пути, имена, секреты заменяются placeholder'ами. Назначение: шаринг шаблона проекта с другими, публикация в community.

`veles import <bundle>` — обратная операция, разворачивает проект в новом инстансе.

## 10. Не-цели

Veles намеренно **не**:
- не тащит в ядро тяжёлые подсистемы (RL/Atropos, batch trajectory, многочисленные gateway-интеграции);
- не строит cloud-инфраструктуру для пользователя (нет SaaS, нет shared backend, нет аккаунтов);
- не объединяет память между проектами автоматически (только manual export/import);
- не сканирует ничего за пределами активного CWD и user-global `~/.veles`; данные других проектов в одной сессии не видит;
- не выполняет деструктивные или сетевые pushes без подтверждения;
- не навязывает LLM Wiki как единственный layout — пользователь свободен описать собственную организацию контента в AGENTS.md;
- не позиционируется как замена IDE — Veles живёт рядом с редактором, не вместо.

## 11. Лицензия и тех-философия

- **Лицензия**: Apache 2.0 с patent grant.
- **Local-first**: всё хранилище локально; cloud — opt-in через отдельные модули.
- **Open interfaces**: провайдер-протокол, формат скиллов и tools, hook-points модулей, агент-протокол — публичные и стабильные внутри major-версии.
- **Стек**: Python 3.13+, SQLite+FTS5 для проектной памяти, Apache-совместимые зависимости. Rust/PyO3 — только для подтверждённых bottleneck'ов.
